# Handoff — разработчику AI-бота `@crmapiprofit_bot`

**Дата:** 2026-04-16
**Владелец:** Денис
**Куда слать правки:** обсуждать в Telegram с Денисом

## Что изменилось

На проде бот врал: на вопрос «кто сегодня на смене?» возвращал выдуманных
сотрудников (Алексей — BMW Vitalik, Сергей — Garcia Bathroom и т.д.).
Причина — это были примеры из старого system prompt, которые модель
воспроизводила дословно вместо реальных вызовов API.

В репозитории `garkorcom/profit-step` мы переписали prompt + зафиксировали
50 канонических use-cases + выложили доки на отдельной публичной странице.
**На твоей стороне ничего не обновилось — бот продолжает отвечать по старому
prompt.**

## Что нужно сделать

### 1. Заменить system prompt

Скачай новый prompt: **https://profit-step.web.app/bot-docs/prompt.md**

Заменяет текущий целиком. Ключевые изменения:
- Убраны все конкретно-выглядящие имена из примеров (`<placeholder>`-токены вместо `Алексей`/`Сергей`).
- Добавлен жёсткий блок `ГЛАВНОЕ ПРАВИЛО: НИКАКИХ ВЫДУМОК` сразу после identity.
- Добавлена секция `АНТИ-ГАЛЛЮЦИНАЦИИ`: все данные должны приходить из API-ответа в текущем ходу разговора, не из памяти модели.
- Добавлен changelog в шапке — при следующем обновлении просто инкремент версии.

### 2. Свериться с use-cases

**https://profit-step.web.app/bot-docs/use-cases.md** — 50 канонических
сценариев. Для каждого указано:
- Что спрашивает пользователь (триггер-фразы)
- Какой endpoint вызывать и в каком порядке
- Что подтвердить у пользователя перед деструктивной операцией
- Шаблон ответа с `<placeholder>` токенами
- Типичные ошибки / HTTP error table

Если twoй бот уже поддерживает tool use / function calling — use-cases
должны лечь 1-в-1 на твой mapping. Если вдруг что-то не покрыто, скажи,
добавим в следующий апдейт.

### 3. OpenAPI spec

**https://profit-step.web.app/api/docs/spec.json** — актуальный OpenAPI.
Автогенерится из реального кода `agentApi`, всегда свежий.

Auth — Bearer token (Master Token). Для действий от лица конкретного
сотрудника — заголовок `X-Impersonate-User: <uid>`.

### 4. Canary-проверка после обновления

Мы хотели бы убедиться, что fix реально доехал. После того как задеплоишь
новый prompt, дай знать — я запущу следующий сценарий и покажу результат:

- **Запрос:** «кто сейчас на смене?»
- **Ожидание:** бот делает реальный вызов `GET /api/time-tracking?status=active` и отвечает по его результатам (возможно — «сейчас никого на смене»). НЕ должен перечислять выдуманных `Алексей / Сергей / Вова`.
- **Если** в ответе появились имена, которых в API-ответе нет → fix не применился.

Если удобнее — могу добавить автоматический canary endpoint на нашей стороне,
который раз в час дёргает твоего бота с контрольным запросом и алертит, если
ответ содержит имена не из списка активных сотрудников. Скажи, если нужно —
сделаем.

## Canary endpoint (опциональный, если хочешь автоматику)

Если хотим автомат, а не ручную проверку, план такой (~2-3 часа работы):

1. Добавить `functions/src/agent/routes/botCanary.ts` с эндпоинтом
   `POST /api/bot-canary/ping`. Принимает webhook от бота с текстом ответа
   на контрольный вопрос, сравнивает со списком реально активных
   `work_sessions`, записывает результат в коллекцию `bot_canary_runs`.

2. Scheduled Cloud Function `runBotCanary` (каждый час в рабочее время):
   отправляет боту контрольный вопрос через Telegram API или напрямую
   через бота-разработчика, ловит ответ, вызывает `/api/bot-canary/ping`.

3. Алерт в Telegram (Денису) если > 2 fails подряд.

Фаза 1 (ручная проверка по шаблону выше) — достаточна для начала. Автомат —
если после фикса бот будет снова дрифтовать или prompt обновляется часто.
