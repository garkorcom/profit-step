## 1. Текущее состояние модуля

| Проблема | Детали |

| Поля связи (relatedTaskId, relatedClientId) есть в типах, но **не используются в UI** | Списание не привязывается к задаче/клиенту через интерфейс |
| Нет push-уведомлений о низких остатках | `getLowStockItems()` есть, но нет алертов |

| Нет прогнозирования потребности | Нет автозаказа |

## 2. Доработки — Фаза 1 (приоритет: высокий)

### 2.1 Привязка списания к задаче и клиенту в UI

**Цель:** Каждое списание материала должно быть привязано к конкретной задаче и клиенту.

**Требования:**
- При создании транзакции типа `write_off`, `transfer` (на объект), `tool_issue` — обязательный выбор клиента и задачи
- Выпадающий список клиентов с поиском
- Выпадающий список задач клиента (фильтрация по выбранному клиенту)
- Сохранение `relatedTaskId`, `relatedClientId` в транзакции
- В журнале транзакций — отображение колонок "Клиент" и "Задача"
- Фильтрация транзакций по клиенту и задаче

**Где менять:**
- `src/features/inventory/` — форма создания транзакции
- `src/pages/inventory/InventoryPage.tsx` — таблица транзакций
- `functions/src/agent/routes/inventory.ts` — валидация на бэкенде

---

### 2.2 Уведомления о низких остатках

**Цель:** Ответственный получает уведомление, когда остаток товара падает ниже `minStock`.

**Требования:**
- После каждой исходящей транзакции — проверка: `totalStock <= minStock`
- Если порог пробит:
  - Push-уведомление в приложении (badge на иконке склада)
  - Уведомление в Telegram admin-боту
  - Запись в коллекцию `notifications`
- Дашборд: виджет "Требуют закупки" — список позиций ниже minStock с кнопкой "Сформировать закупку"
- Настройка: кому отправлять уведомления (admin / manager / конкретный user)

**Где менять:**
- `functions/src/agent/routes/inventory.ts` — триггер после транзакции
- Новый компонент: `LowStockAlert.tsx`
- `src/pages/inventory/InventoryPage.tsx` — виджет на дашборде

- Итоги по каждой локации (суммарная стоимость)

**Где менять:**

### 2.4 Расширенный журнал транзакций

**Цель:** Полноценный журнал движения с фильтрацией и группировкой.

**Требования:**
- Фильтры: по товару, локации, типу операции, периоду (дата от/до), клиенту, задаче, кто выполнил
- Группировка: по дням / неделям / месяцам
- Итоги: количество операций, суммарная стоимость по типам (приход / расход)
- Цветовая индикация: приход (зеленый), расход (красный), перемещение (синий)
- Поиск по названию товара и примечанию (note)
- Пагинация (сейчас limitCount — нужно полноценное)

**Цель:** Полноценный интерфейс для складов типа "vehicle" — машины бригад.

**Требования:**

## 3. Доработки — Фаза 2 (приоритет: средний)

### 3.1 Интеграция "Смета → Склад"

**Цель:** Автоматическое создание плановой потребности в материалах из утвержденной сметы.

**Требования:**
- При утверждении сметы (status: `approved`):
  - Автосоздание `TaskMaterial[]` для задачи с `source: 'estimate'`
  - Каждая позиция сметы типа `material` → `TaskMaterial` с `plannedPrice`, `qty`, `unit`
  - Маппинг позиции сметы на `catalogItemId` (по SKU или ручной выбор)
- При конвертации сметы в задачу — материалы уже привязаны
- UI: в деталях задачи — вкладка "Материалы" с позициями из сметы

**Бизнес-логика:**
```
Смета утверждена
  → Для каждого material-item:
    → Поиск в inventory_catalog по SKU или имени
    → Если найден → создать TaskMaterial с catalogItemId
    → Если не найден → создать TaskMaterial без catalogItemId, status: 'need_purchase'
  → Автоматический подсчет: что есть на складе, что нужно докупить
```

---

### 3.2 План-факт контроль расхода материалов

**Цель:** Сравнение планового расхода (по смете) с фактическим (по транзакциям списания).

**Требования:**
- Отчет по проекту/задаче:
  - Таблица: Материал | Ед. | План (из сметы) | Факт (из транзакций) | Отклонение | Отклонение %
  - Цветовая индикация: зеленый (<10%), желтый (10-25%), красный (>25%)
  - Итоги: плановая стоимость, фактическая стоимость, экономия/перерасход
- Алерты:
  - При списании: если фактический расход превысил плановый на >15% — предупреждение
  - Блокировка списания сверх плана (опционально, настраивается)
- Доступ: admin видит все, manager — свои проекты

**Формула:**
```
Отклонение = (Факт - План) / План × 100%
```

---

### 3.3 Автоматическое формирование закупочного списка

**Цель:** На основании плановой потребности и текущих остатков — автоформирование "что купить".

**Требования:**
- Источники потребности:
  - TaskMaterial со статусом `planned` или `need_purchase`
  - Позиции ниже `minStock`
  - Активные резервирования (`inventory_reservations` со статусом `reserved`)
- Расчет:
  ```
  К закупке = Потребность - Остаток на складе - Активные резервы (от других задач)
  ```
- UI: страница "Закупки" с группировкой по поставщику (если указан)
- Действия: "Создать Purchase Order", "Отправить список в Telegram"
- Связь: `TaskMaterial.shoppingListId` → группировка

---

### 3.4 Списание через Telegram-бот (для бригадиров)

**Цель:** Бригадир на объекте может списать материал через Telegram без входа в CRM.

**Требования:**
- Команда в рабочем боте: `/materials` или `/списать`
- Сценарий:
  1. Бот спрашивает: "Какой материал?" → поиск по каталогу (fuzzy)
  2. "Сколько?" → количество
  3. "Откуда?" → выбор локации (машина / склад)
  4. Подтверждение → создание транзакции `write_off`
  5. Автопривязка к активной сессии работника (clientId, taskId)
- Бот присылает итог дня: "Сегодня списано: ..."

---

### 3.5 Расширенный учет инструмента

**Цель:** Полный жизненный цикл инструмента от закупки до списания.

**Требования:**
- Карточка инструмента:
  - Серийный номер / инвентарный номер
  - Дата покупки, стоимость, гарантия до
  - Текущее местоположение и ответственный
  - Журнал выдачи/возврата (кто, когда, в каком состоянии)
  - Фото при выдаче и возврате
- Плановое ТО:
  - Периодичность (раз в N дней / часов работы)
  - Уведомление о необходимости ТО
- Амортизация:
  - Распределение стоимости инструмента по проектам (пропорционально использованию)
- UI: отдельная вкладка "Инструменты" в модуле склада

## 4. Доработки — Фаза 3 (приоритет: низкий)

### 4.1 Аналитика и отчеты

| Отчет | Описание |
|-------|----------|
| Динамика цен | График изменения закупочных цен по позициям за период |
| Топ-10 по расходу | Самые расходуемые позиции за период (в количестве и деньгах) |
| Оборачиваемость | Как быстро расходуются запасы (дни оборота по каждой позиции) |
| Расход по бригадам | Сколько материалов израсходовал каждый работник/бригада |
| ABC-анализ | A (80% стоимости) / B (15%) / C (5%) — для фокуса закупок |
| Себестоимость проекта | Материалы + труд + подряд = полная себестоимость по проекту |
| Экспорт | CSV / Excel по всем отчетам |

### 4.2 Сканирование QR/штрих-кодов

**Требования:**
- Генерация QR-кода для каждой позиции каталога
- Печать этикеток (формат для принтера этикеток)
- Сканирование камерой телефона → быстрый приход / списание
- Сканирование в Telegram-боте (фото штрих-кода → распознавание)

### 4.3 Учет сроков годности

**Требования:**
- Поле `expirationDate` в `InventoryCatalogItem` (для расходников: герметики, клеи, краски)
- Партионный учет: один товар — несколько партий с разными сроками
- Уведомление за N дней до истечения
- Автоматическое списание просроченных (или алерт для ручного решения)
- FIFO-логика: при списании — сначала товары с ближайшим сроком

### 4.4 Интеграция с поставщиками

**Требования:**
- Справочник поставщиков: название, контакты, условия, история заказов
- Привязка позиций каталога к поставщикам (один товар — несколько поставщиков)
- Сравнение цен поставщиков
- Формирование заказа поставщику из закупочного списка
- История закупок по поставщику

### 4.5 Прогнозирование потребности

**Требования:**
- На основании истории проектов — прогноз потребности в материалах для нового проекта аналогичного типа
- Шаблоны материалов по типу работ (например, "стандартная электрика 2-комнатная квартира")
- AI-подсказка: "Для такого проекта обычно нужно: ..."

## 7. Критерии приемки

### Фаза 1
- [ ] Каждая транзакция списания привязана к клиенту и задаче через UI
- [ ] При падении остатка ниже minStock — уведомление в приложении и Telegram

- [ ] Утвержденная смета автоматически создает TaskMaterial

- [ ] Бригадир может списать материал через Telegram-бот

- [ ] Все 7 аналитических отчетов доступны
- [ ] QR-коды генерируются и сканируются
- [ ] Партионный учет с FIFO для товаров со сроком годности
- [ ] Справочник поставщиков с историей закупок
- [ ] AI-прогнозирование потребности по типу проекта
