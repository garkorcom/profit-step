# Контекст

> Зачем нужен модуль `tasktotime`, какую проблему решает и почему мы выделяем его в отдельный автономный блок. Это входная точка для понимания всего ТЗ.

## Текущая ситуация

Существующий модуль задач (`/crm/tasks?view=board`) — это **GTD-движок**, выросший из inbox-обработки. Он живёт переплетённым с inventory, time-tracking, payroll, telegram, AI, projects, clients, calendar — **~50 файлов с hardcoded `'gtd_tasks'` строкой**.

## Цель `tasktotime`

Превратить задачу в полноценную **производственную единицу работы** для конструкшн-бизнеса со всем набором атрибутов:

- время (плановое и фактическое)
- деньги (себестоимость, продажная цена, бонусы, штрафы)
- материалы
- инструменты
- контакты
- адрес
- акт выполнения

И всё это — **в автономном модуле**, который можно вынести в отдельный пакет / репо в будущем.

## Принцип миграции

- Старый код **не ломаем** до завершения переноса
- После миграции — переносим в `_archived/`
- Параллельно двух модулей в проде **НЕ будет** — фрилансер оба не потянет, плюс дабл-подписки на коллекцию

## Что важно

`tasktotime` — **не «ещё одна view»**. Это **полная замена** текущего блока задач с:

- переименованием коллекции (`gtd_tasks` → `tasktotime_tasks`)
- единым source-of-truth
- consolidated lifecycle state machine
- единой `<TaskCard>` компонентой для всех contexts
