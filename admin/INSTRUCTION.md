# Категория: Admin (Администрирование)

## Назначение
Специализированная секция для управления системой и командой. Доступна только пользователям с установленным `role: 'admin'`. Содержит системные настройки, реестр пользователей и RBAC (Role-Based Access Control) конфигурации.

## Как работать (Сценарии пользователя)
- **Team**: Просмотр списка всех сотрудников. Приглашение новых (или перехват созданных из Telegram-бота `CreateUserFromBot`). Назначение им ролей (`admin`, `worker`, `viewer`) и ставок оплаты (`hourlyRate`).
- **Companies**: Управление системными профилями компаний. CRM поддерживает многоуровневость (Multi-Tenant) — разные команды/компании могут иметь изолированные данные.
- **Tasks (Admin View)**: Глобальный просмотр **всех** задач в системе, а не только своих (в отличие от обычного модуля Tasks & Work).
- **About / Settings**: Системная инфа.

## Основные Связи (Отношения с другими модулями)
- **Admin ↔ Tasks/Finance**: Параметр `hourlyRate` (Почасовая ставка), установленный в разделе Team, критично важен. Без него тайм-трекер не будет генерировать деньги в Time Tracking -> Finance, и у сотрудника будет выдаваться ошибка (Warn: $0/hr).
- **Admin ↔ Аутентификация**: При входе в систему Firebase Auth проверяет `userProfile.role`. Если роль не `admin`, ссылки не отображаются в `Header.tsx`, а попытка зайти на роут выкинет на главную (Routing Guards/Redirects).
- **Admin ↔ OpenClaw (Агенты)**: Бот может создавать пользователя на лету (`/api/users/create-from-bot`), если ему пишет новый рабочий в Telegram. Этот профиль затем попадает в Admin -> Team для утверждения.

## Как это устроено (Для разработчика)
- **Коллекции Firestore**: 
  - `users`: Профили юзеров. Хранят их Firestore UID, `telegramId`, `role`.
  - `companies`: Компании для Multi-tenant (получение `ownerCompanyId`).
- **Файлы/Компоненты**: 
  - `src/pages/admin/TeamManagementPage.tsx`
  - Маршруты защищены через специальный `ProtectedRoute` (Higher-Order Component).
- **Архитектура**: 
  - Роли проверяются строго на сервере (в Firestore Security Rules и функциях). Пользователь не может отправить POST запрос на смену своей ставки, если у него нет прав.
