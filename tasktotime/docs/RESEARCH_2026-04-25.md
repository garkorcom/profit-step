**Дата:** 2026-04-25

**Аудитория:** Денис

## TL;DR (что забираем в ТЗ)

1. **Gantt с двумя полосками (план vs факт)** — стандарт у Procore, Buildertrend, Smartsheet, TeamGantt. Тонкая baseline-полоска снизу + темная actual сверху. Не изобретать.
2. **Critical path — красный, dependencies — линии-стрелки** между барами; Buildertrend красит синим, Procore/Smartsheet — красным. Toggle on/off обязателен (иначе шум).
3. **«Crew swimlanes» как отдельный термин не существует** в Procore/Buildertrend Gantt. Они делают **resource grouping** (группировка строк по assignee/crew). Чистые swimlane diagrams — это другая визуализация (workflow). Для нас: даём group-by row (по бригаде / по этапу / по комнате).
4. **Weather days** — у Buildertrend это auto-pull из ZIP кода + правила «при дожде сдвигать downstream». Для Tampa в сезон гроз (июнь–сентябрь) это must-have.
5. **Иерархия задач — 2 уровня хватит**. Linear/Notion = 2 уровня (parent + sub-issues). ClickUp даёт 7 уровней — это анти-паттерн, путает пользователей. Брать модель Linear: Task → Sub-task с auto-rollup статуса.
6. **Tree visualization для UI: React Flow (xyflow) + dagre layout**. Это де-факто стандарт 2026, ~100k+ weekly downloads, MIT, активно поддерживается. Для древовидной TOC — `react-d3-tree` или просто MUI `TreeView` (нативный). Cytoscape — overkill для нас (graph theory).
7. **Wiki rollup pattern (новое требование Дениса)** — нет готового tool с этой фичей. Ближайший прецедент: R Markdown child documents (`child = 'sections/example.Rmd'`) и Notion's подход «sub-pages внутри database row». Реализуем сами: parent.wiki = concat(child[i].wiki) с разделителями `## {childTitle}`.

## 1. Construction Gantt — что копируем

| Tool | Что сильно | Что копируем в `tasktotime` |
|---|---|---|
| **Procore Schedule** | Critical path **красным** автоматически. Toggle для % complete / dependencies / critical path. View toggles в верхнем баре. Bars показывают assignee имена прямо на баре. | Toggle-bar (critical / deps / %). Имена исполнителей на самом баре (не сбоку). |
| **Buildertrend Schedule** | Critical path **синим**. Когда predecessor сдвигается — connected tasks **автоматически** двигаются. Weather autopull по ZIP коду. Personal days / holidays оверлеем. | Auto-shift downstream при изменении predecessor (это уже в TZ §1.1 `autoShiftEnabled`). Weather-aware из coords. Holiday overlay (US federal + Florida). |
| **Smartsheet Construction** | Color-coded по фазам. Inspections как milestone-блокеры (нельзя начать drywall пока не пройдена inspection). Baseline сравнение. | Inspections / permits как **milestone diamonds** с hard-block на dependent tasks. |
| **TeamGantt** | Baseline = тонкий бар под основным. Светлый planned, тёмный actual overlay. | Двухполосный визуал, один-в-один. |
| **Houzz Pro** | Mobile-first task management для дизайнеров+ремонтников Tampa-style. Drag schedule с телефона. | UX подсказка: drag-to-reschedule на mobile **обязателен** (бригадиры не у компа). |
| **Fieldwire** | Offline-first для field crews, $39/user. Аннотации на чертежах. | Offline-кэш для рабочих с прорабской функцией (вне scope v1, но architecture не должна блокировать). |

**Что НЕ копируем:**
- Procore полный Schedule tool — слишком тяжёлый, лицензия $100k+/год, у нас другой scale
- Primavera P6 — overkill для residential ($5M проектов), нужен для $500M-объектов
- Microsoft Project — не cloud-native, плохой mobile

### Crew swimlanes — что это на самом деле

**Главное открытие:** «crew swimlane» как функция в construction Gantt = **группировка строк (rows) по resource/assignee** в обычном Gantt. Чистые swimlane diagrams (BPMN-style) — это другая визуализация, для workflows, не для timelines (см. [Traqplan](https://traqplan.com/gantt-chart-vs-swimlane-diagram-which-is-better-for-your-project/)).

Для `tasktotime` это значит:
- В Gantt-view даём **group-by**: project / room / crew / executor / phase
- Каждая группа — collapsible header, в неё попадают bars нужных tasks
- Это покрывает 95% «crew swimlane» use cases без отдельной визуализации

## 2. Notion / ClickUp / Linear — модель иерархии

### Что мы берём от Notion ([source](https://www.notion.com/help/tasks-and-dependencies))

- **Sub-items**: при включении Notion автоматически создаёт relation «Parent tasks» ↔ «Sub-tasks». Многоуровневое.
- **Rollup**: тянет данные из child в parent. Например, parent.completionPct = avg(children.completionPct) — формула, не код.
- **Database templates** с sub-pages — каждая task строка может иметь nested wiki page прямо внутри.
- **Recurring templates** — daily/weekly/monthly/yearly. Ограничение: нельзя вложить template в daily-recurring template.

### Что мы берём от ClickUp ([source](https://clickup.com/hierarchy-guide))

- **Hierarchy:** Workspace > Space > Folder > List > Task > Subtask > Nested Subtask > Checklist (8 уровней!)
- **Vердикт:** это анти-паттерн. Денис правильно держит **2 уровня** (Project → Task). Sub-task как 3-й уровень — норм, но дальше **НЕ нужно** (опыт сообщества подтверждает: пользователи теряются).
- **Что копируем**: ClickUp Box view (Workload by person) — у нас может быть «view: загрузка по исполнителю на неделю».

### Что мы берём от Linear ([source](https://linear.app/docs/parent-and-sub-issues))

- **Sub-issues с auto-rollup**:
  - Когда **все** sub-issues = done, parent автоматически становится done
  - Когда parent = done, оставшиеся sub-issues тоже становятся done
- **Sub-issues наследуют project/cycle** от parent при создании — экономит клики
- **Display options**: показать/скрыть sub-issues, фильтр «только parent / только sub / только with children»

**Это ровно та модель, которую мы хотим.** В TZ нужно явно прописать:
- 2 уровня иерархии (Task + Sub-task)
- Auto-rollup статуса по правилам Linear
- Inheritance от parent при create sub-task (project, dueDate, location, materials defaults)

## 3. Trello + Power-Ups — что не копировать

Trello сам Gantt не умеет — нужны power-ups (**Planyway**, TeamGantt, Elegantt, BigPicture). Subtasks реализуются через checklists или power-up `Card Connectors`.

**Verdict:** Trello-стек = bolt-on решение. Для нашего scope (полный жизненный цикл construction task с money/materials/acceptance act) — недостаточно. **Не копировать архитектурно**, можно подсмотреть UX:
- Planyway: subtasks expandable inline в timeline view (источник: [Planyway Gantt](https://planyway.com/templates/gantt))

## 4. Tree visualization для нашей реализации

### Выбор библиотеки (для tree+DAG view tasks)

| Library | Bundle | Подходит для | Verdict |
|---|---|---|---|
| **React Flow / @xyflow/react** | средне (~100k+ weekly downloads, активно maintained) | Node-based UI, drag-drop, custom nodes, edges, zoom/pan | **РЕКОМЕНДУЮ для DAG зависимостей** |
| **react-d3-tree** | малый | Чистое дерево (parent-child), вертикаль/горизонталь | Подходит для «tree-of-subtasks» view |
| **MUI X TreeView** | уже в нашем bundle (MUI v7) | Простое expand/collapse дерево | **РЕКОМЕНДУЮ для sidebar TOC** |
| **Cytoscape.js** | большой | Graph theory, сложные сети | Overkill для нас |
| **D3 hierarchy** | малый | Низкоуровневый, нужно строить самим | Слишком много кода |
| **GoJS** | большой, **коммерческий** ($) | Enterprise diagramming | Не использовать (платный) |

**Рекомендация:**

1. **Sidebar / TOC дерева задач** (read-only, expand/collapse) → `MUI X TreeView` (нулевой bundle cost)
2. **DAG зависимостей (interactive)** — кружочки задач + стрелки между ними → `@xyflow/react` + `dagre` для auto-layout
3. **Mind map / brainstorm view** (если будет в roadmap) → тоже `@xyflow/react`

Не таскать в bundle 3 разные libs. React Flow + MUI TreeView покрывает 100%.

Sources: [npm-compare graph libs](https://npm-compare.com/cytoscape,d3-graphviz,dagre-d3,gojs,vis-network), [Dagre Tree React Flow example](https://reactflow.dev/examples/layout/dagre).

## 5. Wiki / Knowledge attached to task

### Прецеденты в индустрии

| Pattern | Tool | Описание |
|---|---|---|
| Single task wiki page (inline editor) | Notion sub-pages, ClickUp Docs in Tasks, Linear issue description с rich-text | Markdown editor встроен прямо в task detail |
| Linked wiki documents | Confluence ↔ Jira (`Link Confluence content` button + `Jira Issues Macro`) | Внешний документ, привязан bi-directional |
| Templates (recurring task) | Notion database templates, Smartsheet templates | Шаблон с pre-filled wiki, materials, checklist |
| Versioning / history | Notion page history, Confluence page history | Built-in, поверх обычного doc storage |

### Wiki rollup (новое требование Дениса) — нет готового решения

Поискал «markdown rollup parent task aggregate child» — **нет существующего tool с этой фичей**. Ближайшие прецеденты:

1. **R Markdown child documents** — синтаксис `child = 'sections/example.Rmd'` в parent, при render всё concat'ится. ([source](https://bookdown.org/yihui/rmarkdown-cookbook/child-document.html))
2. **Smartsheet parent rollup** — но только для числовых полей, не для текста / markdown.
3. **ServiceNow parent-child task relationships** — статус roll-up, не контент.

**Вывод:** реализуем сами. Простая модель:

```ts
// Псевдо-код
function aggregateWiki(parent: Task): string {
  const ownContent = parent.wiki ?? '';
  const childWikis = parent.subtaskIds
    .map(id => loadTask(id))
    .filter(t => t.wiki?.trim())
    .map(t => `\n\n---\n\n## ${t.taskNumber} — ${t.title}\n\n${t.wiki}`);
  return ownContent + childWikis.join('');
}
```

**Расширение:** при render parent wiki — кнопка «Включить rollup из sub-tasks» (toggle), а не автоматический concat. Иначе будут дубли при экспорте акта.

## 6. Construction-specific knowledge / project memory

| Tool | Что для контекста |
|---|---|
| **Procore Project Files** | Централизованный file storage по project, link на task |
| **Buildertrend Documents + Daily Logs** | Daily log с фото/погодой/часами. Daily logs — **отдельная сущность**, не attachment к task |
| **Procore Punch List** ([source](https://www.procore.com/library/construction-punch-list-explained)) | Отдельный модуль, появляется в конце project, items имеют responsible party + due date. Привязаны к Schedule через статус «зависит от punch closeout». |

**Вывод для `tasktotime`:**
- Daily Log — отдельная фича позже (не в task), но в API оставить hook для будущего
- Punch List items — это просто `Task` с `category = 'punch'` + `phase = 'closeout'`. Не отдельная сущность. Это сразу даёт reuse: deps, money, materials, acceptance act работают для punch items без специального кода.

## 8. Concrete recommendations для `TZ_TASKTOTIME.md` v0.2

### Must-have (добавить в spec)

1. **§Gantt view spec:**
   - Plan vs Actual = две полоски (тонкая baseline + тёмная actual overlay), TeamGantt-стиль
   - Critical path toggle (red), dependencies toggle (lines), % complete toggle
   - Group-by selector: `none / project / room / crew / executor / phase` (это и есть наш «swimlane»)
   - Drag-to-reschedule на mobile (Houzz Pro precedent)
   - Auto-shift downstream при изменении predecessor (уже в TZ как `autoShiftEnabled`)
   - Weather day overlay для Tampa (use lat/lng из task.location, pull NOAA для прогноза/факта; в дождь — visual marker «☂», правило для auto-shift = опционально, чтобы не сдвигать молча)
   - Milestone diamonds для permits / inspections / client approvals — hard-block на dependent tasks

2. **§Hierarchy:**
   - Жёстко **2 уровня** (Task → Sub-task). Не плодить ClickUp-style 7-этажный hierarchy.
   - Linear-style auto-rollup: parent.lifecycle = 'completed' когда все subs = 'completed'. Reverse: при completion parent — спросить «закрыть всех subs?» (не молча).
   - Sub-task наследует от parent: `projectId, location, dueAt (default = parent.dueAt - 1d), assignedTo, materials defaults`
   - Display filter: `all / only parents / only with children / only orphan subs`

3. **§Wiki:**
   - Inline markdown editor в task detail (как Linear/Notion)
   - **Wiki rollup** (новое): toggle-кнопка «Show aggregated wiki» — concat parent + всех children с разделителями `## {childNumber} — {childTitle}`. Use case: экспорт акта по большой задаче.
   - Templates: recurring task templates (Notion-style), для повторяющихся работ типа «недельная инспекция site»
   - Versioning: snapshot при каждом save (Firestore subcollection `wiki_versions`)

4. **§Tree visualization:**
   - Sidebar TOC: MUI X TreeView (zero new bundle cost)
   - DAG view зависимостей: @xyflow/react + dagre (новая dep, ~middle-weight)
   - НЕ добавлять Cytoscape / D3-tree / GoJS

5. **§Punch list integration:**
   - НЕ отдельная сущность, просто `Task` с `category = 'punch' | 'main'` и `phase = 'closeout'`
   - Reuse всего Task contract (deps, money, materials, acceptance)

### Nice-to-have (фейс-2)

- Mobile offline cache (Fieldwire precedent)
- Daily Log как отдельная сущность с привязкой к task (Buildertrend)
- ClickUp Box view = «загрузка по исполнителю на неделю» (heatmap)

### Анти-паттерны (явно записать «НЕ делать»)

- Не делать 7-уровневую иерархию (ClickUp anti-pattern)
- Не делать swimlane как отдельную визуализацию (group-by покрывает 95%)
- Не делать автоматический wiki rollup (опасно для актов — нужен toggle)
- Не таскать в bundle более 1 graph viz library
- Не auto-shifting weather days молча — нужно user confirmation для каждой задачи
