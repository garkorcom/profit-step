🔧 InfoDebag · как устроена доработка и тест

Каждая страница — живая спецификация.

Сама себя документирует. Сама собирает фидбек. Сама знает где сломана. AI триажит раз в час и двигает backlog без ручного Jira.

👀 Смотреть на живом примере → 🗂 Все страницы прототипа 📋 Полный ТЗ

🤔 Зачем вообще

Три боли которые решаем — они звучат очевидно, но в SMB реально всегда так:

1. Тесты живут отдельно от продукта

QA пишет кейсы в Google Docs или TestRail. Через 3 месяца: кейсы устарели, никто их не прогоняет, продукт живёт своей жизнью. Баги находят клиенты, не тесты.

2. Фидбек теряется в Slack / стикерах / DM

«Денис, тут кнопка не работает», через час забыли. Нет единого backlog'а. Бригадир шлёт баг в чат, админ кликает и забывает.

3. ТЗ устаревает в день deploy'а

Написали Confluence-страницу, задеплоили. Через неделю добавили поле — ТЗ не обновили. Через месяц уже непонятно что было задумано а что случайно появилось.

Наше решение: каждая страница несёт своё ТЗ + свои тесты + свой фидбек внутри себя. Всё data-driven, всё синхронизировано, AI держит в актуальном состоянии.

🔄 Как это работает — data flow

Что происходит от момента когда юзер жмёт 🐛 до момента когда agent пишет patch:

Цикл 🐛 → AI → fix → UC зелёный. Юзер видит результат на той же странице, где оставил фидбек.
1
Юзер на странице жмёт 🐛 Пишет «approve кнопка не реагирует» + severity=high → submit. Автоматом приложится: URL · viewport · live DOM snapshot · lastClick selector (в плане).
2
Запись летит в STORAGE.feedback Сейчас = localStorage (прототип). В проде = Firestore pages/{pageId}/feedback/{id}. FAB badge обновляется.
3
AI-Triage-агент просыпается раз в час (Claude) dup-detection · severity re-class · classifies bug / feature / regression · предлагает patch-diff · if blocker → push в Telegram owner'у страницы.
4
Запись попадает в _feedback.html Kanban 4 колонки: raw → triaged → in scope → done. Drag-n-drop или автоматом после триажа.
5
Если bug связан с UC — UC помечается 🔴 failing Counter на странице обновляется. Coverage падает. Master TZ перестраивает roadmap.
6
GTD-task создаётся для PAGES[id].owner Если owner = finance-admin-agent — агент патчит автоматом. Если owner = живой человек — в его inbox.
7
После fix'а — AI-runner прогоняет UC Claude Vision открывает страницу в headless, сверяет с expected. 🔴 → 🟢. Audit event записывается. Цикл закрыт.

🧬 Данные — три типа

Всё живёт в одном meta.js. Три главных типа:

PAGES — каталог страниц портала. UseCase прикреплён к странице и хранит тесты. Feedback тоже прикреплён к странице и ссылается на failing UC.

1. PAGES — спецификация страницы (статично, в коде)

// meta.js
PAGES['expenses'] = {
  id:      'expenses',
  title:   'Затраты',
  owner:   'finance-admin-agent',    // кому идут tickets
  purpose: 'Админский поток одобрения чеков...',
  inputs:  [{ name:'expenses[]', from:'ExpenseService', required:true }],
  outputs: [{ name:'approve', to:'ExpenseService.approve', trigger:'✓ button' }],
  features:   ['KPI: approved/pending/billable', ...],      // Характеристики
  advantages: ['Админ видит что ждёт одобрения сразу', ...], // Преимущества
  benefits:   ['Approvals за 30 сек вместо 5 мин', ...],    // Выгоды
  agents:  ['receipt-ocr-agent', 'approval-bot'],
  apis:    ['POST /api/expenses/:id/approve'],
}

2. UseCase — acceptance test (живёт в localStorage / Firestore)

{
  id:             'expenses-uc-001',
  pageId:         'expenses',
  title:          'Админ одобряет expense < $500 одним кликом',
  role:           'admin',                 // admin|foreman|worker|client|system
  preconditions:  ['авторизован как admin', 'в очереди 1+ submitted'],
  steps: [
    { action:'открыть /expenses', expected:'очередь approval видна' },
    { action:'нажать ✓',          expected:'chip меняется на approved' },
  ],
  status:         'passing',  // passing|failing|untested|flaky|draft|spec-only
  source:         'manual',   // manual|voice|ai-generated|from-bug
  lastRun:        '2026-04-21T00:26:50Z',
  failureReason:  null,
  linkedBugs:     [],       // feedback ids, если failing
  linkedAgents:   [],       // кто участвовал
  globalRef:      'UC-47',  // ссылка в USE_CASES.md
}
6 статусов. Главный цикл: 🟣 → ⚪ → 🟢 ↔ 🔴. Regression = пошло обратно. Spec-only = «UC есть, фичи нет» (roadmap).

3. Feedback — баг/улучшение (живёт в localStorage / Firestore)

{
  id:            'fb-1744...000',
  pageId:        'expenses',
  type:          'bug',                 // bug|improvement
  severity:      'high',                // low|med|high|blocker
  text:          'approve кнопка не реагирует',
  createdBy:     'Денис',
  createdAt:     '2026-04-21T...',
  status:        'raw',                // raw|triaged|in-scope|done

  // ─── автоматически прикладывается (в плане) ───
  context: {
    url:          '/expenses.html',
    viewport:     { w:1280, h:800 },
    liveDom:      '...outerHTML on click...',    // DOM на момент бага
    sourceFile:   '...fetch /expenses.html...',    // оригинал из git
    lastClick:    { selector:'.btn.primary' },   // что нажимал
    screenshot:   'data:image/png;base64,...',    // html2canvas
    console:      ['Warning: ...'],               // intercepted errors
  },

  // ─── поля после триажа (AI Claude) ───
  triagedBy:     'claude-opus-4.7',
  suggestedFix:  'diff --git a/expenses.html...',
  linkedUseCase: 'expenses-uc-004',
}

Попробуй прогнать UC

Нажми ▶ Прогнать → confirm-диалог. OK = passed, Cancel = failed (попросит причину). Статус 🟢/🔴 обновляется, counter наверху пересчитывается.

4

Добавь свой UC

Нажми + Добавить → форма. Title + role + preconditions (по одному на строку) + steps (формат action | expected на строку). Сохраняется сразу в localStorage.

5

Оставь фидбек через 🐛

Плавающая кнопка справа снизу. Клик → drawer. Тип (💡 Улучшить / 🐛 Не работает) · severity · описание → Сохранить. Badge на FAB показывает сколько фидбеков накопилось. Таб «История» показывает все записи.

6

Превратить UC в баг или в ТЗ

Раскрой failing-UC → 🐛 В баг: создаёт feedback-запись + UC помечается failing. 📝 В ТЗ: UC → spec-only, попадает в roadmap. 🗑 Удалить: soft delete с reason.

💭 Открытые решения

То что обсуждается до имплементации:

🔬 Code-snapshot в debug-bar

Идея: при submit фидбека автоматически прикладывать document.documentElement.outerHTML + fetch(location.pathname) (оригинал из git) + lastClick selector. AI-triage получает ready-to-patch пакет.

вариант A: live DOM вариант B: source file вариант C: element picker вариант D: всё сразу

🎙 Voice input для UC

«Как админ я хочу чтобы при одобрении больше 500 долларов система спрашивала 2 раза» → Claude парсит в {role, title, preconditions, steps}. Web Speech API (Chrome) или Whisper API (Safari fallback).

🤖 AI-Triage pipeline

Cron раз в час → проходит raw фидбек → dup-detection (embeddings similarity) → severity re-class → patch suggestion → if blocker → Telegram push owner'у. Celery / Cloud Scheduler / raw cron?

📊 Coverage map

Master TZ показывает heatmap: страницы × роли, цвет = coverage. Сразу видно «клиент-портал не тестируется совсем», «onboarding тестирует только админ». Actionable.
