## 1. Цели и бизнес-обоснование

### 1.1. Проблемы текущей CRM

- 79% клиентов (19 из 24) находятся в статусе «забытые» — нет системы касаний
- Карточка клиента показывает справочную информацию (контакты), но не даёт сквозной картины: нет истории сделок, встреч, коммуникаций, фактической маржи
- Отсутствуют сущности Сделка (Deal), Встреча (Meeting), Этап производства, Субподрядчик
- Задачи и сметы не связаны: после выигрыша работы переносятся в производство вручную
- Нет учёта фактической себестоимости: план/факт по работам и материалам не считается
- Нет интеграции с мессенджерами, хотя клиентская коммуникация идёт в Telegram
- Нет клиентского портала — клиент не видит прогресс и не может оплатить онлайн

### 1.2. Целевые бизнес-эффекты

| Метрика | Текущее | Целевое (6 мес) |
|---|---|---|
| Доля «забытых» клиентов | 79% | ≤ 25% |
| Средняя маржа проекта | базовая | +5–8 п.п. |
| Время от акта до оплаты | — | ≤ 7 дней |
| NPS клиентов | — | ≥ 55 |
| LTV клиента | базовый | +20–30% |
| Экономия от тендеров субподряда | 0% | 8–15% |
| Repeat business rate | — | ≥ 30% |

### 1.3. Принципы дизайна системы

- **Сквозная связность:** клиент → сделка → смета → проект → производство → акт — без ручных переносов
- **Двухслойные сметы:** отдельная клиентская (цена) и внутренняя (себестоимость)
- **План/факт везде:** каждая запланированная цифра сверяется с фактической
- **Автоматизация рутины:** автозаполнение, автосоздание папок, автогенерация счетов по триггерам
- **Прозрачность для клиента, защита внутренних данных от субчиков**
- **AI как помощник, человек принимает решения** (human-in-the-loop для критичных действий)

## 2. Глоссарий и модель данных

### 2.1. Ключевые сущности

| Сущность | Назначение |
|---|---|
| **Client** | Клиент компании, вечная сущность, хранит контакты, реквизиты, LTV |
| **Deal** | Сделка — этап продажи, одна или несколько за жизнь клиента |
| **Meeting** | Встреча (замер, презентация, приёмка), привязана к Deal или Project |
| **Estimate (Client)** | Смета для клиента — цена продажи |
| **Estimate (Internal)** | Внутренняя смета — себестоимость и план производства |
| **Project** | Проект — физическая реализация выигранной сделки |
| **Task** | Задача в проекте с флагами `billable`, `production` |
| **ProductionItem** | Позиция производственного плана с план/факт по времени и материалам |
| **TimeEntry** | Запись сессии таймера сотрудника |
| **MaterialPurchase** | Закупка материалов с чеком |
| **Tender** | Тендер на субподряд, создаётся из Internal Estimate |
| **Bid** | Предложение субчика по тендеру |
| **Subcontractor** | Справочник субподрядчиков с рейтингом и историей |
| **SubcontractorContract** | Договор с выбранным субчиком |
| **Invoice** | Счёт клиенту |
| **Act** | Акт выполненных работ |
| **ChangeOrder** | Допсоглашение по проекту |
| **File/Folder** | Файлы с автосозданной иерархией папок |
| **TelegramMessage** | Сообщение из привязанного чата |

### 2.2. Иерархия связей

```
CLIENT
 ├── DEAL[] ──── MEETING[], ESTIMATE_CLIENT[] (версии)
 │      └── (если выигран) → PROJECT
 │
 ├── PROJECT[]
 │      ├── TASK[] ─── FILES[]
 │      ├── ESTIMATE_CLIENT (одобренный) ─── INVOICE[]
 │      ├── ESTIMATE_INTERNAL ─── TENDER[] ─── BID[] ─── CONTRACT
 │      ├── PRODUCTION_ITEM[] ─── TIME_ENTRY[], MATERIAL_PURCHASE[]
 │      ├── MEETING[]
 │      ├── CHANGE_ORDER[]
 │      └── ACT[]
 │
 ├── COMMUNICATIONS (Telegram, Email, Calls, Notes)
 ├── FILES/FOLDERS (автоструктура)
 └── FINANCE (LTV, Balance, Margin — агрегированное)
```

## 3. Сквозной бизнес-процесс

```
1. Лид → Client + Deal «Первичное обращение»
2. Deal: Meeting «Замер» → Estimate (Client) v1, v2…
3. Deal: «Выиграна» → автосоздание Project из одобренной сметы
4. Project: автозаполнение адреса и реквизитов из Client
5. Project: добавление Tasks, автосоздание файловой структуры
6. Генерация Estimate (Client) из задач → отправка → одобрение
7. Выставление Invoice (аванс)
8. Генерация Estimate (Internal) из клиентской → расчёт себестоимости
9. По позициям с `sourcing = tender` → создание Tender → приглашение субчиков
10. Сбор Bids → Bid Comparison → выбор победителя → Contract
11. Перевод позиций в ProductionItem (in-house или subcontractor)
12. Производство: таймеры сотрудников, закупки материалов, фотоотчёты
13. Завершение этапа → Act → оплата клиентом → следующий этап
14. Финальный Act → Reconciliation (план/факт) → P&L проекта
15. Сервисная фаза: гарантия, ТО, NPS, реферальная программа
```

## 4. Модуль 1. Карточка клиента v2

### 4.1. Шапка карточки (sticky при скролле)

**Строка 1:** логотип клиента, имя (inline-edit), бейдж lifecycle stage (Lead / Prospect / Active / Repeat / Churned / VIP), бейдж сегмента (A/B/C/VIP), health score (круговой индикатор 0–100), churn risk icon, кнопка «⭐ избранное».

**Строка 2 — 8 KPI-плашек:** Баланс, LTV, Маржа ($ и %), Активных сделок, Активных проектов, Открытых/просроченных задач, Ближайшая встреча, Последний контакт (дней назад).

**Строка 3 — быстрые действия:** Позвонить, Telegram, Email, Встреча, Сделка, Смета, Проект, Начисление, Заметка, AI-ассистент.

### 4.2. Вкладки

1. **Обзор** — сводный дашборд (воронка сделок клиента, текущие проекты, встречи, финансы, AI-инсайты, документы)
2. **Сделки** — все Deals с Kanban активных и списком архивных
3. **Сметы** — клиентские сметы всех сделок, версии, статусы (Sent/Viewed/Approved)
4. **Проекты** — карточки проектов, прогресс, доступ к производству
5. **Встречи** — календарь + список с типами и outcome
6. **Производство** — сквозной Gantt по всем активным проектам клиента
7. **Финансы** — P&L по клиенту, Cash Flow, журнал транзакций, счета
8. **Коммуникации** — унифицированная лента Telegram + Email + Звонки + Заметки
9. **Файлы** — дерево папок клиента (все проекты, все задачи)
10. **AI-ассистент** — чат с агентом в контексте клиента

### 4.3. Правый sidebar (sticky)

Next Best Action от AI, ближайшая встреча, просроченные задачи, ответственные, быстрая заметка (Enter = сохранить), таймер до красной зоны, связанные клиенты (реферер, рефералы).

### 4.4. Поля Client (расширенные)

Базовые (имя, тип, email, phone, address, website), расширенные (`lifecycle_stage`, `segment`, `source`, `referral_by_client_id`, `preferred_channel`, `preferred_language`, `timezone`, `tax_info`, `billing_info`, `payment_terms`, `tax_rate`, `currency`, `decision_makers[]`, `tags[]`, `nps_score`, `health_score`, `churn_risk`, `ltv`, `total_margin`, `avg_payment_delay_days`).

### 5.1. Сущность Deal

Поля: `client_id`, `title`, `pipeline_id`, `stage` (Новая / Замер назначен / Замер выполнен / КП в работе / КП отправлено / Переговоры / Выиграна / Проиграна), `expected_value`, `probability`, `expected_close_date`, `actual_close_date`, `lost_reason` (обязательно при проигрыше), `source`, `owner_id`, `estimates[]`, `meetings[]`, `project_id` (после выигрыша).

### 5.2. Воронка (Sales Pipeline)

- Подсветка «застрявших» сделок (нет активности N дней)

### 5.3. Сущность Meeting

Поля: `client_id`, `deal_id?`, `project_id?`, `type` (Первый контакт / Замер / Презентация КП / Подписание / Выезд / Приёмка этапа / Финальная сдача / Сервис), `status`, `start_at`, `end_at`, `location`, `attendees[]`, `client_attendees[]`, `agenda`, `outcome` (обязательно после проведения), `next_steps`, `calendar_event_id`.

- Календарь (день/неделя/месяц)
- Интеграция с Google Calendar и Outlook (двусторонняя синхронизация)
- Автонапоминания клиенту в Telegram (24ч и 2ч до встречи)
- Обязательное заполнение `outcome` после встречи (иначе блокировка перехода сделки дальше)
- AI-транскрипция звонков (при интеграции с телефонией)

## 6. Модуль 3. Проекты, задачи, файлы

### 6.1. Автозаполнение при создании Project

При `POST /projects` с `client_id` автоматически копируются из клиента:
- `address` (с возможностью override через чекбокс «Адрес объекта отличается»)
- `billing_info` (название, ИНН/EIN, банк, счёт, адрес выставления)
- `contact_person`, `phone`, `email`
- `tax_rate`, `payment_terms`, `currency`, `preferred_language`, `timezone`
- `project_manager` = ответственный менеджер клиента

Данные хранятся как **snapshot** в Project — изменение клиента не меняет историю проектов.

Форма создания в 3 шага: Основное → Контакты и реквизиты → План работ (опционально).

### 6.2. Сущность Task

Поля: `project_id`, `title`, `description`, `status` (To Do / In Progress / Blocked / Done / Accepted), `priority`, `assignee_id`, `due_date`, `estimated_hours`, `actual_hours`, `position`, `parent_task_id`, `checklist[]`, `files[]`, `folder_path`, `labels[]`, `billable` (boolean — попадёт в клиентскую смету), `production` (boolean — попадёт в производство), `estimated_price_client`, `estimated_cost_internal`, `unit`, `quantity`, `rate`.

### 6.3. UI задач

- Быстрое добавление строкой
- Массовый импорт из шаблона
- AI-подсказка типовых задач
- Drag-and-drop сортировка и группировка по этапам
- Прикрепление файлов drag-and-drop

### 6.4. Файловая структура (автосоздание)

```
/Clients/{client_code}-{client_name}/
  /_client-docs/                              ← общие документы клиента
  /_communications/
  /{project_code}-{project_name}/             ← папка проекта (автосоздание)
    /_project-docs/                           ← договор, КП, акты
    /_photos/
    /_invoices/
    /_materials/
    /tasks/
      /{task_code}-{task_name}/               ← папка задачи (автосоздание)
        /inputs/
        /work/
        /outputs/
```

### 6.5. Требования к хранилищу

- S3-совместимое хранилище (AWS S3 / Cloudflare R2 / MinIO)
- Версионирование на уровне бакета
- Автогенерация превью (image, PDF, video)
- Полнотекстовый поиск по содержимому PDF/Docs
- Опциональная интеграция с Google Drive / Dropbox
- Права наследуются от проекта; клиент видит только файлы с `visible_to_client = true`

## 7. Модуль 4. Сметы (клиентская и внутренняя)

### 7.1. Client Estimate

**Генерация из задач:**
- Кнопка «Создать смету из задач» → конструктор
- Слева — задачи с `billable = true`, справа — черновик сметы
- Объединение нескольких задач в одну позицию
- Добавление позиций не из задач (материалы, доставка)

**Структура:** Header (номер, даты, реквизиты) → Sections (разделы) → Line Items (с `source_task_id`, `price`, `quantity`, `unit`) → Subtotal / Discount / Tax / Total → Terms → Signature.

**Функции:**
- Версионирование (v1, v2, v3)
- PDF с брендингом компании
- Отправка через Email / Telegram / публичную ссылку с трекингом открытий
- Онлайн-одобрение клиентом
- Сравнение версий
- Внутренний блок маржи (только для менеджера): сумма / себестоимость / маржа с цветовой подсветкой

### 7.2. Internal Estimate (Production Plan)

**Генерация:** кнопка «Сгенерировать производственный план» из одобренной Client Estimate.

**Структура ProductionItem:** `source_estimate_item_id`, `client_price` (справочно), plan-блок (`planned_labor_hours`, `planned_labor_cost`, `planned_materials[]`, `planned_materials_cost`, `planned_subcontractor_cost`, `planned_overhead`, `planned_total_cost`), fact-блок (agreegates из TimeEntry и MaterialPurchase), variance-блок, `sourcing_strategy` (in_house / tender / direct_award), `assignee`, `status`.

**Справочники для расчёта:**
- Ставки сотрудников ($/час)
- Ставки бригад ($/день)
- Каталог материалов (наименование, ед.изм., цена, поставщик)
- Накладные расходы (% или фикс)

**UI:** таблица позиций с раскрытием в детальное планирование. Сводка: Total client price / Total planned cost / Planned margin $ и %.

## 8. Модуль 5. Производство и тайм-трекинг

### 8.1. Mobile / PWA для сотрудника

- Список назначенных ProductionItem на сегодня
- Кнопка **«▶ Старт»** на каждой позиции
- Во время работы — большой видимый таймер, кнопки «⏸ Пауза» / «⏹ Стоп»
- При остановке — запрос процента выполнения, комментария, фото прогресса
- Переключение между позициями останавливает старый таймер, запускает новый

### 8.2. TimeEntry

Поля: `production_item_id`, `employee_id`, `started_at`, `ended_at`, `duration_minutes`, `hourly_rate` (snapshot), `cost`, `notes`, `geo_location` (опц.), `photos[]`, `approved_by_pm`.

Автообновление при каждом Entry: `production_item.actual_labor_hours += duration`, `actual_labor_cost += cost`, пересчёт variance.

### 8.3. Материалы

**MaterialPurchase:** привязка к ProductionItem, поставщик, дата, сумма, фото чека (с OCR-распознаванием позиций), статус (Planned / Ordered / Delivered / Used).

**Мобильный сценарий:** бригадир в магазине → фото чека → AI распознал → привязал к ProductionItem → расход зафиксирован.

### 8.4. Дашборд прораба «Today»

- Кто работает сейчас (таймеры в реальном времени)
- Кто и сколько работал сегодня
- Превышение плановых часов (красные флажки)
- Закрытые этапы за день
- Алерты: на 50% работ израсходовано 70% часов → предупреждение

### 8.5. Субподряд на ProductionItem

Поля: `assigned_type` (brigade / subcontractor), `subcontractor_id`, `subcontractor_contract_id`, `subcontractor_bid_price`, `internal_planned_cost`, `variance_vs_plan`.

## 9. Модуль 6. Тендеры и субподряд

### 9.1. Справочник Subcontractor

Поля: `company_name`, `contact_person`, `phone`, `email`, `telegram_username`, `specializations[]`, `coverage_area[]`, `tax_info`, `documents[]` (договор, страховка, лицензии), `rating` (авто 1–5), `stats` (win_rate, on_time_rate, avg_variance, quality_score, total_revenue), `tags[]` (VIP / рискованный / надёжный / дешёвый), `blacklisted`, `notes`.

### 9.2. Создание Tender из Internal Estimate

Визард в 3 шага:
1. Выбор позиций (не все) → scope без цен
2. Выбор субчиков (AI-подбор по специализации/региону/рейтингу)
3. Дедлайны, описание, файлы (ТЗ, чертежи, фото), visibility (Private / Open)

Цены Internal Estimate **автоматически вырезаются** из того, что увидит субчик.

### 9.3. Портал субподрядчика

Публичная страница по токену (без регистрации):

**Субчик видит:** название тендера, регион/адрес (можно анонимизировать), scope работ с объёмами без цен, файлы, сроки, Q&A, форма bid.

**Субчик НЕ видит:** имя и контакты клиента, бюджет, цены других субчиков, внутреннюю себестоимость, список других приглашённых (опция).

**Форма Bid:** цена за единицу по каждой позиции, комментарии, исключения, альтернативы, срок старта, длительность, гарантия, условия оплаты, приложения. Редактирование до дедлайна, после — блокировка.

### 9.4. Bid Comparison Matrix

Таблица с колонками: Позиция / Объём / Субчик А / Субчик Б / … / Наш план (internal cost).

Функции:
- Подсветка лучшей цены по позиции
- AI-алерты выбросов (слишком дёшево → не понял объём; слишком дорого → попытка заработать)
- Weighted scoring (настраиваемые веса: цена 50% / сроки 20% / рейтинг 20% / гарантия 10%)
- Split Award (взять часть у одного, часть у другого)
- Negotiation Rounds (Best and Final Offer)

### 9.5. Awarding → Contract

- Кнопка «Award» на выигравшем Bid + обязательный `award_reason`
- Автоуведомление проигравшим (вежливый отказ)
- Генерация SubcontractorContract (PDF) → онлайн-подпись
- Связанные ProductionItems получают `subcontractor_id`
- Обновление `Subcontractor.stats.total_bids_won`

### 9.6. Контроль исполнения

Субчик в своём портале: отчёты прогресса, Payment Requests, загрузка фото.
Менеджер: утверждение оплат, QC по позициям, фиксация отклонений от bid.
По завершении: акт с субчиком → сверка bid vs факт → оценка 1–5 → автоматическое обновление рейтинга.

### 9.7. Безопасность

- Невозможность редактирования bid после дедлайна
- Лог всех изменений
- Водяные знаки на PDF с ID субчика (для выявления утечки)
- NDA-чекбокс при первой подаче
- Только после Award субчик получает контакты клиента

## 10. Модуль 7. Финансы, акты, сверка

### 10.1. Invoice

Генерация из Estimate → копирование позиций → выбор (полный / частичный / процент / фикс-сумма авансом) → автозаполнение реквизитов → PDF → отправка → онлайн-оплата (Stripe / ЮKassa / ручная отметка).

### 10.2. График платежей

Настраивается в проекте (пример: 30% аванс, 40% черновой этап, 30% сдача). Каждая позиция графика — триггер автогенерации Invoice при подписании соответствующего этапа клиентом.

### 10.3. Act of Completion

Генерируется при закрытии всех ProductionItems этапа: список выполненных позиций, ссылки на фото, сумма → PDF → онлайн-подпись клиентом → триггер следующего Invoice.

### 10.4. Reconciliation (Сверка)

Экран по проекту с таблицей план/факт:

| Категория | План | Факт | Отклонение | Статус |
|---|---|---|---|---|
| Работы (часы) | | | | |
| Работы ($) | | | | |
| Материалы | | | | |
| Субподряд | | | | |
| Накладные | | | | |
| **ИТОГО COST** | | | | |
| Клиенту (Revenue) | | | | |
| **МАРЖА $ и %** | | | | |

Колонка «Сверка оплат»: выставлено / оплачено клиентом / выплачено бригаде / оплачено поставщикам / Cash Flow проекта.

### 10.5. P&L по клиенту и по проекту

Автоматический расчёт в реальном времени: Revenue − Cost of Materials − Cost of Labor − Cost of Subcontractors − Overheads = Margin.

### 10.6. Change Orders

Создание в процессе проекта → одобрение клиентом → пересчёт Project value, Internal Estimate, plans и margin.

## 11. Модуль 8. Telegram-интеграция

### 11.1. Архитектура

Telegram Bot API + Telegram Client API (MTProto) для групп. Webhook-сервер в Profit Step, очередь (Redis), таблица `telegram_messages` с привязкой к `client_id`, `project_id`, `deal_id`.

### 11.2. Уровни чатов

- **Chat клиента (общий)** — привязан к Client
- **Chat проекта** — привязан к Project (менеджер + ПМ + бригадир + клиент + прораб клиента)
- **Chat сделки** — временный на период продажи
- **Chat ЛПР** — личный для конфиденциальных вопросов
- **Chat субподрядчика** — отдельный бот для субчиков

### 11.3. Привязка к сущности

Кнопка «Привязать Telegram-чат» → 3 варианта:
1. Добавить бота в существующую группу → ввод кода верификации
2. Бот создаёт новую группу с участниками
3. Личный чат: клиент пишет `/start CODE`

Сохраняется `chat_id`, `chat_title`, `chat_type`, `role`.

### 11.4. Функции переписки в CRM

- Unified inbox вкладка в карточке клиента
- История из Telegram при первой загрузке, далее real-time через webhook
- Отправка: текст, файлы, голос, фото, документы
- Форматирование, цитирование, реакции, редактирование (в рамках API)
- Поиск по истории клиента
- Индикация «кто печатает» (через Userbot)

### 11.5. Автоматизация

- Автопарсинг сообщений AI: создание задачи / Change Order / эскалация жалобы
- Еженедельное AI-саммари переписки
- Автоперевод иностранных сообщений
- Напоминания о встречах клиенту
- Отправка фотоотчётов с объекта
- Счета и NPS-опросы

### 11.6. Разграничение доступа

Менеджер — только свои клиенты. Клиент не видит внутренних заметок. Админ — всё. Логирование всех отправок. Флаг `hide_from_client_portal` для приватных чатов.

## 12. Модуль 9. Клиентский портал

### 12.1. Концепция

`client.profit-step.app/{token}` — без регистрации, magic-link из email/Telegram.

### 12.2. Что видит клиент по фазам

**На стадии Deal:** статус в воронке, запланированные встречи с картой, присланные сметы с возможностью одобрить онлайн, чат с менеджером.

**На стадии Project:**
- Gantt этапов с прогрессом
- Фотоотчёты по дням
- Бригада на объекте (фото, имена)
- Документы (договор, смета, акты)
- График платежей + онлайн-оплата
- Запрос Change Order от клиента
- Чат с ПМ и бригадиром

**На стадии Service:** гарантийный паспорт, вызов мастера, история обслуживаний, персональные предложения.

### 12.3. Чего клиент НЕ видит

Внутренние заметки, себестоимость, маржу, зарплаты бригады, переписку менеджеров, других клиентов, internal estimate, тендеры, bids субчиков.

### 12.4. Upsell и рост прибыли

- Онлайн-подпись акта → ускорение оплат на 30–50%
- Запрос Change Order самим клиентом → +15–25% к марже проекта
